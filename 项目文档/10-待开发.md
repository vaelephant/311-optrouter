2. 三层路由含义

Layer 1：粗路由

目标是：

先用几乎 0 成本的方法，拦住大部分明显请求。

这里不做深理解，只看这些：
	•	当前消息长度
	•	是否有代码块
	•	是否有 JSON / SQL / 报错栈
	•	是否要求“详细分析 / 完整代码 / 分步推理”
	•	历史轮数
	•	附件数量
	•	是否是新会话
	•	用户选择的模式：eco / balanced / premium

这一层和 ClawRouter 很像：
重点是快、便宜、本地完成。ClawRouter 的路由核心也是本地多维启发式评分，再映射到 tier/model。

⸻

Layer 2：上下文路由

这是 OptRouter 比 ClawRouter 更值得做强的一层。

因为很多问题像：
	•	“继续改一下”
	•	“按刚才那个来”
	•	“这个为什么不行”
	•	“就按上次的方式改”

单看当前一句，复杂度判断会失真。
所以第二层不去读全量上下文，而是读一个持续维护的会话摘要。

例如：

{
  "topic": "Next.js + FastAPI 聊天系统",
  "mode": "code_refactor",
  "recent_files": ["api/chat/route.ts", "rag.py"],
  "history_depth": 12,
  "current_goal": "继续修复流式输出中断问题",
  "risk_level": "medium",
  "preferred_quality": "balanced"
}

这一层解决的是：

“当前问题在这个会话里，实际上属于什么任务强度。”

⸻

Layer 3：精细路由

只有当前两层仍然模糊时，才调用一个小模型做低成本复判。

也就是：
	•	不是每个请求都先分类一次
	•	只有“模糊区间”才触发小模型
	•	输入也不是全部上下文，而是：
	•	当前问题
	•	最近几轮摘要
	•	会话 summary
	•	结构化特征

这样分类成本很小，但准确率会明显更高。

4. 对应到 OptRouter 的模型层

你可以把最终路由到的目标先抽象成这些产品档位，而不是直接暴露底层模型名：

block:

- balanced    -> 默认均衡
- premium     -> 最高质量

reasoning -> 强推理模型
longctx   -> 超长上下文模型

这样产品层会更清晰。

5. 最关键的一点：会话摘要怎么维护

OptRouter 真正要强的地方，不是“更复杂的规则”，而是：

维护一个便宜、稳定、持续更新的 Session Summary。

建议每轮回答后，顺手更新：
	•	当前主题 topic
	•	当前任务类型 task_type
	•	当前目标 current_goal
	•	关键文件 / 关键对象
	•	风险等级 risk_level
	•	用户偏好 quality_preference
	•	会话阶段 stage

例如：

type SessionSummary = {
  topic: string
  taskType: "chat" | "code" | "analysis" | "writing" | "planning"
  currentGoal: string
  recentArtifacts: string[]
  historyDepth: number
  riskLevel: "low" | "medium" | "high"
  preferredMode: "eco" | "balanced" | "premium"
  lastChosenTier?: string
}

6. 一个简化版伪代码

function route(request, sessionSummary) {
  const basicFeatures = extractBasicFeatures(request)

  const coarse = coarseRoute(basicFeatures)
  if (coarse.confidence > 0.8) {
    return coarse.targetTier
  }

  const contextFeatures = mergeWithSessionSummary(
    basicFeatures,
    sessionSummary
  )

  const contextual = contextualRoute(contextFeatures)
  if (contextual.confidence > 0.75) {
    return contextual.targetTier
  }

  const refined = smallModelClassify({
    message: request.message,
    sessionSummary,
    features: contextFeatures
  })

  return refined.targetTier
}

7. 和 ClawRouter 的差异，一眼看懂版

ClawRouter:
当前 prompt + 本地规则  --> 直接粗分流

OptRouter:
当前 prompt + 本地规则
        ↓
   会话摘要增强
        ↓
   小模型复判兜底
        ↓
     最终路由

所以你可以把它理解成：

ClawRouter 更像“快而省”的单层启发式路由。
OptRouter 可以升级成“粗分流 + 上下文感知 + 模糊复判”的三层路由。

一、整体思路

这个架构可以分成 6 个核心层：

1. 接入层

用户、应用、Agent、企业系统都先打到统一入口：
	•	OpenAI-compatible API
	•	SDK
	•	Agent 调用
	•	企业内部系统接入

这层的目标是：

先把多模型能力变成一个统一入口。

⸻

2. Router Core

这是 OptRouter 最核心的引擎。

它不只是“把请求转发出去”，而是负责：
	•	判断请求强度
	•	判断上下文依赖
	•	判断用户偏好
	•	判断成本/质量/延迟要求
	•	最终选择哪个模型更合适

这里就是你前面说的重点：

LLM 聚合 + 智能路由

⸻

3. Policy Engine

路由不能只靠“复杂度”这一个维度。
真正工程里，要综合策略。

所以需要一个策略层：
	•	Cost Policy：成本优先
	•	Quality Policy：质量优先
	•	Latency Policy：速度优先
	•	Risk Policy：高风险任务更保守
	•	User Mode：用户指定 eco / balanced / premium

这层的作用是：

把路由从“技术判断”升级成“产品决策”。

⸻

4. Model Registry + Provider Adapters

这一层是 LLM 聚合的基础。

要做两件事：

Model Registry
维护每个模型的信息：
	•	模型名
	•	提供商
	•	输入/输出价格
	•	最大上下文
	•	擅长领域
	•	延迟表现
	•	稳定性评分
	•	是否支持 function calling / json mode / streaming

Provider Adapter
把不同厂商统一适配掉：
	•	OpenAI 参数格式
	•	Anthropic 格式
	•	Gemini 格式
	•	DeepSeek 格式
	•	其他模型厂商格式

这样上层 Router 就不用关心各家接口差异。

⸻

5. Fallback / Retry / Cache

这是平台可用性的核心。

Fallback
如果目标模型失败：
	•	自动切下一个备选模型
	•	或同档位替代模型
	•	或降级到安全默认模型

Retry
	•	短时网络错误自动重试
	•	限流后退避
	•	超时兜底

Cache
	•	重复请求缓存
	•	标准 Prompt 缓存
	•	相似请求缓存
	•	工具结果缓存

这一层解决的是：

稳定性和成本。

⸻

6. Session / Billing / Observability

这一层是平台化的关键。

Session Summary Updater
每轮对话结束后更新：
	•	topic
	•	currentGoal
	•	taskType
	•	recentArtifacts
	•	historyDepth
	•	riskLevel
	•	preferredMode

Billing / Usage Metering
记录：
	•	调用哪个模型
	•	输入/输出 token
	•	成本多少
	•	路由前后节省多少
	•	fallback 发生几次

Observability
监控：
	•	请求量
	•	平均延迟
	•	成功率
	•	模型命中率
	•	路由分布
	•	成本变化

四、最核心的 5 个对象

type RequestProfile = {
  messageLength: number
  historyDepth: number
  hasCode: boolean
  hasJson: boolean
  hasLongContext: boolean

2. SessionSummary

描述当前会话状态。

type SessionSummary = {
  topic: string
  currentGoal: string
  taskType: string
  recentArtifacts: string[]
  historyDepth: number
  riskLevel: "low" | "medium" | "high"
  preferredMode: "eco" | "balanced" | "premium"

3. ModelCapability

描述每个模型的能力画像。

type ModelCapability = {

  outputCost: number

  strengths: string[]

  supportsTools: boolean
  supportsVision: boolean

最终路由结果。

type RoutingDecision = {
