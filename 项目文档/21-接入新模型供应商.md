# 快速接入新模型供应商

分两种：**OpenAI 兼容 API**（Together / Groq / Azure 等）和 **非兼容协议**（需自己实现请求/响应）。

---

## 一、OpenAI 兼容的供应商（最快）

协议与 OpenAI `POST /v1/chat/completions` 一致（请求体、响应 JSON、流式 SSE 格式相同）。  
只需**新增一个供应商独立文件**（如 `together.rs`），包装 `OpenAIProvider`，再完成下面 4 步即可。

### 步骤 1：增加 Provider 枚举

**文件：** `gateway/src/router/model_router.rs`

- 在 `enum ProviderType` 里加一项，例如：`Together`。
- 在 `impl ProviderType` 的 `as_str()` 里加对应字符串，例如：`ProviderType::Together => "together"`。

### 步骤 2：配置 Key 和 Base URL

**文件：** `gateway/src/config/providers.rs`

- 在 `ProviderConfig` 里增加字段，例如：`together_api_key: Option<String>`、`together_base_url: String`。
- 在 `from_env()` 里从环境变量读取，例如：`TOGETHER_API_KEY`、`TOGETHER_BASE_URL`（默认 `https://api.together.xyz/v1`）。

**文件：** `gateway/src/config/mod.rs`（若存在单独 config 入口）

- 在 `api_key_for()` 的 match 里加上新 Provider 分支，返回 `self.providers.together_api_key.as_deref()`。
- 若有 `base_url_for()`，同样加新分支返回 `self.providers.together_base_url`。

### 步骤 3：为供应商建独立 Provider 文件

**文件：** `gateway/src/providers/mod.rs`

- 增加 `pub mod xxx;`
- 在 `build_provider()` 的 match 里加分支，例如：`ProviderType::Together => Box::new(together::TogetherProvider::new(api_key, provider_url))`。

**新建文件：** `gateway/src/providers/xxx.rs`（如 `together.rs`、`ollama.rs`）

- **OpenAI 兼容**：定义结构体包装 `openai::OpenAIProvider`，实现 `Provider` 并委托给 `inner` 即可（见 `together.rs`）。
- **非兼容**：在該文件内完整实现 `call()` 与 `convert_response()`。

### 步骤 4：登记模型与路由

**文件：** `gateway/src/router/registry.rs`

- 在 `build_default_registry()` 里为供应商的模型名建 `RouteInfo`，例如：
  - `model`: 对外暴露的模型 ID（如 `meta-llama/Llama-3.1-70B-Instruct-Turbo` 或你自定义的短名）。
  - `provider_url`: 该供应商的 Base URL（如 `https://api.together.xyz/v1`）。
  - `provider`: `ProviderType::Together`。
  - `fallback_*`: 按需填写，不需要就 `None`。

### 步骤 5（必做）：定价与可用模型

- **DB**：在 `model_pricing` 表插入新模型的记录（`model_name`、`input_price`、`output_price`、`provider`、`enabled = true`），否则 `GET /v1/models` 不会列出、chat 会报 model 不可用。
- **环境变量**：部署时配置 `TOGETHER_API_KEY`、按需配置 `TOGETHER_BASE_URL`。

## 二、非 OpenAI 协议的供应商

请求/响应格式与 OpenAI 不同（例如 Anthropic、Google 的原始 API），需要**单独实现** `Provider` trait。

### 步骤 1～2：同上

- 在 `ProviderType`、配置（api_key、base_url）、`api_key_for` / `base_url_for` 里加入新供应商。

### 步骤 3：新建 Provider 实现

**新建文件：** `gateway/src/providers/xxx.rs`（如 `together.rs` 或供应商名）

- 定义结构体，保存 `api_key`、`base_url`（若需要）。
- 实现 `Provider` trait：
  - `call()`：用 reqwest 发 HTTP 请求，把 `ChatCompletionRequest` 转成该供应商的请求体，把响应原样返回 `reqwest::Response`（流式即 SSE 流）。
  - `convert_response()`：把供应商的**非流式** JSON 响应转成 `ChatCompletionResponse`（字段对齐 `protocol/openai.rs`）。
- 流式：若供应商的 SSE 与 OpenAI 不同，需在 `proxy` 层或本层解析 usage；当前 OpenAI 兼容流可直接复用现有流式 + usage 截取逻辑。

**在 `gateway/src/providers/mod.rs`：**

- 增加 `pub mod xxx;`
- 在 `build_provider()` 里对新 `ProviderType` 返回 `Box::new(xxx::XxxProvider::new(api_key, provider_url))`。

### 步骤 4～5：同上

- 在 `registry.rs` 登记模型与 URL。
- 在 `model_pricing` 加定价，配置环境变量。

## 三、清单速查（OpenAI 兼容）

| 步骤 | 文件 | 动作 |
|------|------|------|
| 1 | `router/model_router.rs` | `ProviderType` + `as_str()` |
| 2 | `config/providers.rs` | 字段 + `from_env()` |
| 2 | `config/mod.rs` | `api_key_for()`（及 `base_url_for()`） |
| 3 | `providers/mod.rs` | `pub mod xxx` + `build_provider()` 新分支 |

| 5 | DB + 环境变量 | `model_pricing` 插入行，配置 `XXX_API_KEY` 等 |

## 四、示例：Together AI（已接入）

本仓库已按「一」完成 Together 接入，可作为复制模板：

| 步骤 | 修改位置 |
|------|----------|
| 1 | `gateway/src/router/model_router.rs`：`ProviderType::Together`、`as_str() => "together"` |
| 2 | `gateway/src/config/providers.rs`：`together_api_key`、`together_base_url` + `from_env()` |
| 2 | `gateway/src/config/mod.rs`：`api_key_for(Together)`、`base_url_for(Together)` |
| 3 | `gateway/src/providers/mod.rs`：`pub mod together`，`ProviderType::Together` → `together::TogetherProvider::new(...)` |
| 3 | `gateway/src/providers/together.rs`：独立文件，包装 `OpenAIProvider` |
| 4 | `gateway/src/router/registry.rs`：Together 模型列表（如 Llama、Mixtral） |

**你还需要：**

- 在 `model_pricing` 表插入 Together 模型定价（`model_name` 与 registry 中一致、`provider = 'together'`、`enabled = true`），否则 `GET /v1/models` 不会返回、chat 会判为不可用。
- 设置环境变量：`TOGETHER_API_KEY=...`；可选 `TOGETHER_BASE_URL`（默认 `https://api.together.xyz/v1`）。

接入其它 OpenAI 兼容供应商（Groq、Anyscale、Azure OpenAI 等）：复制上述 1～4 步，把 `Together` 换成新枚举（如 `Groq`）、配置与 registry 中的 URL/模型名改为该供应商即可。

## 五、示例：Ollama（已接入）

本仓库已接入 Ollama（本地/自托管），协议与 OpenAI 兼容，**通常无需 API Key**（未配置 `OLLAMA_API_KEY` 时使用空字符串）。

| 步骤 | 修改位置 |
|------|----------|
| 1 | `gateway/src/router/model_router.rs`：`ProviderType::Ollama`、`as_str() => "ollama"` |
| 2 | `gateway/src/config/providers.rs`：`ollama_api_key`（可选）、`ollama_base_url`（默认 `http://localhost:11434/v1`） |
| 2 | `gateway/src/config/mod.rs`：`api_key_for(Ollama)`、`base_url_for(Ollama)` |
| 3 | `gateway/src/providers/mod.rs`：`pub mod ollama`，`ProviderType::Ollama` → `ollama::OllamaProvider::new(...)` |
| 3 | `gateway/src/providers/ollama.rs`：独立文件，包装 `OpenAIProvider` |
| 4 | `gateway/src/router/registry.rs`：Ollama 模型列表（如 `llama3.2`、`mistral`、`qwen2.5` 等） |
| — | `gateway/src/api/chat.rs`：Ollama 且 `api_key_for` 为 `None` 时使用 `""`，不报错 |

**使用：**

- 本地启动 Ollama 后，可选设置 `OLLAMA_BASE_URL`（默认 `http://localhost:11434/v1`），可不设置 `OLLAMA_API_KEY`。
- 在 `model_pricing` 表插入 Ollama 模型定价（`provider = 'ollama'`、`enabled = true`），否则 `GET /v1/models` 不会列出。

## 六、如何测试 Ollama

Gateway 已启动的前提下，按下面步骤即可验证 Ollama 是否可用。

### 1. 确保 Ollama 在跑

- 本机：终端执行 `ollama serve`（若已安装 Ollama，通常已在后台运行）。
- 拉一个已登记的模型，例如：`ollama pull llama3.2`。
- 若 Ollama 不在本机，设置环境变量：`OLLAMA_BASE_URL=http://你的主机:11434/v1`。

### 2. 数据库准备

- **API Key**：请求必须带合法 Key。在 `api_keys` 表里需有一条 `status = 'active'` 的记录，且请求头里的 Bearer 明文 Key 的 SHA-256 等于该行的 `key_hash`（一般由 Web 端创建 Key 时写入）。
- **模型定价**：在 `model_pricing` 表插入要测的 Ollama 模型，且 `enabled = true`。例如（表需有 `provider`、`enabled` 列时）：

```sql
INSERT INTO model_pricing (model_name, input_price, output_price, provider, enabled)
VALUES ('llama3.2', 0, 0, 'ollama', true)
ON CONFLICT (model_name) DO UPDATE SET enabled = true, provider = 'ollama';
```

Registry 里已登记的 Ollama 模型名包括：`llama3.2`、`llama3.2:latest`、`llama3.1`、`llama3.1:latest`、`mistral`、`qwen2.5`、`gemma2`、`phi3`。`model_name` 需与之一致。

### 3. 用 curl 调 chat 接口

Gateway 默认端口为 `3001`（可通过 `PORT` 环境变量修改）。把 `YOUR_API_KEY` 换成你在库里有的、对应的明文 API Key。

**非流式：**

```bash
curl -X POST http://localhost:3001/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "llama3.2",
    "messages": [{"role": "user", "content": "说你好"}],
    "stream": false
  }'
```

**流式：**

```bash
curl -X POST http://localhost:3001/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "llama3.2",
    "messages": [{"role": "user", "content": "说你好"}],
    "stream": true
  }'
```

### 4. 如何判断「Ollama 是否可用」

- 返回 200 且 JSON 里有 `choices[0].message.content`（非流式）或 SSE 里有正常 content 增量（流式）→ 说明 Gateway 已把请求正确转发到 Ollama，Ollama 可用。
- 401 → 检查 API Key 和 `api_keys` 表。
- 400 且提示 `model 'xxx' is not available` → 检查 `model_pricing` 是否已插入该模型且 `enabled = true`。
- 502/超时 → 检查 Ollama 是否在跑、`OLLAMA_BASE_URL` 是否可达（本机默认 `http://localhost:11434/v1`）。

可选：先查模型列表，确认网关能读到定价表里的 Ollama 模型：

```bash
curl -s http://localhost:3001/v1/models -H "Authorization: Bearer YOUR_API_KEY"
```
