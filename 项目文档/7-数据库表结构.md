# 7-数据库表结构

## 📋 概述

本文档详细说明当前项目的数据库表结构，基于 Prisma Schema 定义。

## 🗄️ 数据库配置

**数据库类型：** PostgreSQL  
**ORM：** Prisma 7.x  
**配置文件：** `prisma/schema.prisma`

**重要说明：**
- Prisma 7.x 要求 `datasource` 中不包含 `url` 配置
- `url` 通过环境变量 `DATABASE_URL` 提供
- 使用 `@prisma/adapter-pg` 和 `pg` 连接池

## 📊 表结构总览

### 核心业务表

1. **users** - 用户表
2. **api_keys** - API Key 表
3. **usage_logs** - 使用记录表
4. **model_pricing** - 模型定价表

### 财务相关表

5. **user_balances** - 用户余额表
6. **transactions** - 财务流水表
7. **plans** - 套餐定义表
8. **subscriptions** - 用户订阅表

### 统计相关表

9. **user_login_logs** - 用户登录日志表
10. **user_behavior_logs** - 用户行为日志表

### 邀请系统表

11. **invite_code** - 邀请码表
12. **invite_relation** - 邀请关系表
13. **invite_reward_rule** - 邀请奖励规则表
14. **invite_reward_record** - 用户邀请奖励记录表

## 📋 详细表结构

### 1. users 表

**表名：** `users`  
**用途：** 系统用户账户

**字段：**

| 字段名 | 类型 | 说明 | 约束 |
|--------|------|------|------|
| id | UUID | 用户唯一ID | 主键，默认 uuid() |
| email | String | 登录邮箱 | 唯一索引 |
| password | String? | 密码哈希（bcrypt） | 可为空 |
| role | String | 用户角色 | 默认 "user" |
| created_at | DateTime | 创建时间 | 默认 now() |
| updated_at | DateTime | 更新时间 | 自动更新 |

**关联关系：**
- `apiKeys` → `api_keys` 表（一对多）
- `usageLogs` → `usage_logs` 表（一对多）
- `transactions` → `transactions` 表（一对多）
- `balance` → `user_balances` 表（一对一）
- `subscriptions` → `subscriptions` 表（一对多）
- `loginLogs` → `user_login_logs` 表（一对多）
- `behaviorLogs` → `user_behavior_logs` 表（一对多）
- `inviteCodes` → `invite_code` 表（一对多）
- `inviteRelationsAsInvitee` → `invite_relation` 表（作为被邀请人）
- `inviteRelationsAsInviter` → `invite_relation` 表（作为邀请人）
- `inviteRewardRecords` → `invite_reward_record` 表（一对多）

**索引：**
- `email`（唯一索引）

### 2. api_keys 表

**表名：** `api_keys`  
**用途：** API Key 管理（OpenAI 风格认证）

**字段：**

| 字段名 | 类型 | 说明 | 约束 |
|--------|------|------|------|
| id | UUID | Key 唯一ID | 主键，默认 uuid() |
| user_id | UUID | 所属用户ID | 外键，关联 users |
| key_hash | String | 密钥哈希（SHA256） | 不存明文 |
| name | String? | Key 名称 | 可为空 |
| status | ApiKeyStatus | 状态 | 默认 active |
| rate_limit_per_min | Int | 每分钟请求限制 | 默认 60 |
| created_at | DateTime | 创建时间 | 默认 now() |
| last_used_at | DateTime? | 最后使用时间 | 可为空 |

**枚举类型：** `ApiKeyStatus`
- `active` - 激活
- `disabled` - 已禁用
- `expired` - 已过期

**关联关系：**
- `user` → `users` 表（多对一）
- `usageLogs` → `usage_logs` 表（一对多）

**索引：**
- `user_id`（索引：`idx_api_keys_user`）

### 3. usage_logs 表

## 📝 数据类型说明

### UUID

- **用途：** 用户ID、API Key ID、订阅ID等
- **格式：** `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- **生成：** Prisma 默认使用 `uuid()` 函数

### Decimal

- **用途：** 金额、价格、余额等财务数据
- **精度：** 
  - `Decimal(10,6)` - 价格和费用（6位小数）
  - `Decimal(12,4)` - 余额和流水（4位小数）
- **原因：** 防止浮点数精度误差

### DateTime

- **用途：** 时间戳字段
- **格式：** ISO 8601（`2024-01-01T10:00:00Z`）
- **时区：** UTC（数据库存储）

### String (ISO 格式)

- **用途：** 邀请系统的时间字段（`expires_at`, `created_at` 等）
- **格式：** ISO 8601 字符串
- **原因：** 兼容原有数据结构

## 🔍 索引策略

### 主键索引

所有表都有主键索引（自动创建）

### 外键索引

所有外键字段都有索引（提高关联查询性能）

### 复合索引

- `usage_logs`: `[user_id, created_at]` - 用户使用记录查询
- `transactions`: `[user_id, created_at]` - 用户流水查询
- `subscriptions`: `[user_id, status]` - 用户订阅状态查询

### 业务索引

- `api_keys.user_id` - 用户 API Key 查询
- `user_login_logs.login_at` - 登录时间范围查询
- `user_behavior_logs.function_name` - 功能使用统计
- `invite_code.code` - 邀请码验证（唯一索引）

## 📊 数据统计

### 表大小估算

- **users**: 预计 < 10万条
- **api_keys**: 预计 < 50万条
- **usage_logs**: 预计 > 1000万条（需要定期归档）
- **user_login_logs**: 预计 > 100万条（需要定期归档）
- **user_behavior_logs**: 预计 > 100万条（需要定期归档）

### 性能优化建议

1. **定期归档**
   - `usage_logs`、`user_login_logs`、`user_behavior_logs` 需要定期归档
   - 建议保留最近 90 天的数据，历史数据迁移到归档表

2. **分区表**
   - 对于超大数据量的表，考虑使用 PostgreSQL 分区表
   - 按时间分区（如按月分区）

3. **读写分离**
   - 统计查询使用只读副本
   - 写入操作使用主库

## 🔒 安全考虑

### 密码存储

- 使用 bcryptjs 哈希，不存储明文
- 哈希值存储在 `users.password` 字段

### API Key 存储

- 使用 SHA256 哈希，不存储明文
- 哈希值存储在 `api_keys.key_hash` 字段

### 数据删除

- 使用软删除（通过 `status` 字段标记）
- 或使用级联删除（`onDelete: Cascade`）

## 📝 注意事项

1. **时区处理**
   - 所有 `DateTime` 字段使用 UTC 时间
   - 前端显示时转换为本地时区

2. **Decimal 精度**
   - 财务相关字段使用 `Decimal` 类型
   - 避免使用 `Float` 或 `Double`（精度问题）

3. **UUID vs Integer**
   - 用户相关表使用 UUID（安全性）
   - 邀请系统表使用 Integer（性能）

4. **日期格式**
   - 大部分表使用 `DateTime` 类型
   - 邀请系统表使用 `String`（ISO 格式，兼容原有数据）

5. **索引维护**
   - 定期检查索引使用情况
   - 删除未使用的索引以提高写入性能
