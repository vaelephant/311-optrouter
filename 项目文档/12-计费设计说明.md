# 计费设计说明

## 流程概览

```
用户请求（带 model）→ 网关鉴权 → 按 model 查定价 → 按 model 路由到不同厂商 URL
    → 调用上游 → 拿到 usage（input/output tokens）→ 按该 model 单价算费 → 扣余额并落库
```

- **用户**：只面向你的 API URL，请求里带 `model`（如 `gpt-4o`、`claude-3-5-sonnet`）。
- **网关**：根据 `model` 查定价、路由到对应厂商、用该 model 的单价计费。

不同模型对应不同厂商、不同 URL，计费价格也不同，当前设计就是**按「模型」维度**区分的。

---

## 当前设计（已支持「每个模型价格不同」）

### 1. 定价表：`model_pricing`

| 字段 | 含义 |
|------|------|
| `model_name` | 模型 ID（与请求里的 `model` 一致，主键） |
| `input_price` | 每 1K **输入** token 单价 |
| `output_price` | 每 1K **输出** token 单价 |
| `enabled` | 是否允许调用 |

- 每个模型一行，**不同模型可以设不同** `input_price` / `output_price`。
- 网关里通过 `get_model_pricing(db, request.model)` 按「用户请求的 model」取价。

### 2. 计费公式（`metrics::compute_cost`）

```text
cost = (input_tokens / 1000) × input_price + (output_tokens / 1000) × output_price
```

- 上游返回 `usage.prompt_tokens` / `completion_tokens`，网关用**该次请求用的 model 的单价**算出一笔 `cost`。
- 非流式：响应里拿到 usage 后算费、扣款、写 `usage_logs` + `transactions`。  
- 流式：流结束拿到 usage 再算费、扣款、写库。

### 3. 扣费与记录

- **预检**：调用上游前 `get_user_balance`，余额 ≤ 0 直接 402，不发起请求。
- **扣费**：`bill_in_tx` 内：写 `usage_logs`（model、tokens、cost、latency_ms）、扣 `user_balances`、写 `transactions`，保证一致。

## 设计要点

### 按「请求的 model」计费

- 计费用的是**用户请求里的 `model`** 对应的 `model_pricing` 行，不是「实际调用的 URL」。
- 这样做的含义：**用户为「选择的那款模型」付费**；你内部是直连 OpenAI 还是通过代理，对用户价格一致。

### Fallback 时的计费

- 当前：主模型失败后 fallback 到备用模型（如 gpt-4 → gpt-4o-mini），**仍按请求的 model（gpt-4）的定价**计费。
- 优点：用户价格可预期；缺点：若备用模型更便宜，你多付成本、少收钱。
- 若希望「按实际调用的模型计费」，需要在 fallback 成功后用**实际调用的 model** 再查一次 `model_pricing`，用该 model 的单价算 `cost`，并在 `usage_logs.model` 里记实际模型（或同时记请求 model + 实际 model）。当前实现是「按请求 model 计费」。

### 货币单位统一

- 代码注释里出现过「美元/1K tokens」，而 API 返回给前端是「费用: X **元**」。
- 建议在 `model_pricing` 和全链路统一约定一种货币（例如全部用**元**），并在文档和字段命名上写清楚（如 `input_price_cny` 或注明「单位：元/1K tokens」），避免混用美元/元。

### 定价数据从哪来

- 当前：`get_model_pricing` 已从 Postgres `model_pricing` 表读；`GET /v1/models/{model}/pricing` 在 `api/models.rs` 里仍是静态列表，可改为同样查 `model_pricing` 表，便于前端展示与计费一致。
- 运维：新上一个模型时，在 `model_pricing` 插入一行（model_name、input_price、output_price、enabled）；调价时改对应行即可，无需改代码。

## 小结

| 问题 | 当前做法 |
|------|----------|
| 不同模型不同价？ | 是，`model_pricing` 按 model 各一行，单价不同即可。 |
| 计费依据？ | 用户请求的 `model` + 上游返回的 input/output tokens。 |
| 谁先发生？ | 先查余额（≤0 则 402），再调上游；调完按 usage 算费、扣款、写 usage_logs/transactions。 |
| Fallback？ | 按「请求的 model」计费；若需按「实际调用 model」计费需改 handler 用实际 model 查价。 |

按上述方式，**「用户访问你的 API → 你按请求的 model 访问不同模型 URL，每个模型计费价格不同」** 已经成立；只需保证 `model_pricing` 表与各 model 的单价维护正确，以及货币单位、fallback 策略与产品预期一致即可。

## 企业级升级建议（OptRouter 级架构）

当前设计评分：**架构清晰度 9/10，企业级程度 8/10，扩展能力 7.5/10**。  
以下升级可将整体提升到 **9.5/10**，避免未来在 fallback、利润控制、账单透明度上踩坑。

### 一、Fallback 计费：按实际模型计费 + 双字段记录

**现状问题：** 用户请求 gpt-4、fallback 到 gpt-4o-mini 仍按 gpt-4 计费 → 账单不透明；若请求便宜模型 fallback 到贵模型则平台亏损。

**推荐：** `usage_logs` 增加 **requested_model**、**actual_model**；**cost = actual_model 的单价**。用户透明、成本准确、不亏钱。

### 二、定价表增加 provider

同一模型多来源（openai/gpt-4o、azure/gpt-4o）价格不同。建议：**model_id + provider**（或联合主键），字段增加 **provider**、**currency**。示例：provider=openai, model_name=gpt-4o, input_price, output_price。

### 三、平台加价（利润来源）

**方案 A：** `provider_input_price` / `provider_output_price`（成本）与 `user_input_price` / `user_output_price`（售价）。  
**方案 B：** 成本价 + **markup_rate**（如 1.2 = 20% 加价）。计费用 user 价扣款，用 provider 价算成本与报表。

### 四、余额预检升级：estimated_cost

**现状：** 仅 `balance <= 0` 拒绝。**风险：** 余额 $0.01、本次 $0.2 会透支。  
**建议：** 调用前检查 **balance >= estimated_cost**，估算方式：`max_tokens × output_price_per_1k / 1000`（可加 input 估算）；不足则 402。

### 五、usage_logs 建议字段

增加 **requested_model**、**actual_model**、**provider**、**request_id**，便于对账、按 provider/模型统计、fallback 分析、延迟排查。

### 六、Metrics（企业必备）

在现有 latency 基础上增加：**model_latency_ms**、**provider_latency_ms**、**tokens_per_second**；暴露 Prometheus：`gateway_requests_total`、`gateway_latency_ms`、`model_usage_tokens`、`billing_cost_total`，配合 Grafana 监控。

### 七、推荐完整计费流程（企业版）

验证 API Key → 检查 **balance >= estimated_cost** → 网关 → 路由 → Provider API → 流式代理 → 返回 usage → **cost = tokens × actual_model 单价** → 事务写 usage_logs（含 requested/actual/provider/request_id）、transactions、扣减余额。

### 八、未来扩展：Auto Router

用户请求 `model: auto` 时，网关自动选最快/最便宜/最稳定模型；路由层决定 actual_model，按 actual_model 查价计费；usage_logs 中 requested_model=auto、actual_model=具体模型。

---

## 升级路线小结

| 项目 | 当前 | 建议 |
|------|------|------|
| Fallback 计费 | 按请求 model | 按 **actual_model** 计费，表记 requested_model + actual_model |
| 定价表 | model_name 单维度 | 增加 **provider**、**currency** |
| 利润 | 无显式加价 | **provider 价 + user 价** 或 **markup_rate** |
| 余额预检 | balance ≤ 0 | **balance ≥ estimated_cost** |
| usage_logs | model, tokens, cost | 增加 **requested_model, actual_model, provider, request_id** |
| 监控 | latency | **model/provider 延迟、tokens/s、Prometheus 指标** |

按上述方向迭代，即可升级为 **OptRouter 级别的企业计费与可观测架构**。
