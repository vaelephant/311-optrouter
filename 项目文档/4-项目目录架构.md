## 🔑 核心设计原则

### Route Group 特性

- **Route Group `(name)` 不影响 URL 路径**
  - `app/(auth)/login/page.tsx` → `/login` ✅
  - `app/(admin)/page.tsx` → `/admin` ✅
  - `app/(api-keys)/keys/page.tsx` → `/keys` ✅

### Domain 层规范

每个模块必须包含 `domain/` 目录，包含：
- `*.service.ts` - 业务编排层（业务逻辑）
- `*.repo.ts` - 数据访问层（数据库操作）
- `*.types.ts` - 类型定义

### API Routes 组织

- **模块私有 API** → 放在 `app/(module)/api/` 下
- **全局核心 API** → 放在 `app/api/` 下（如 `/api/chat`, `/api/models`）

### 组件组织

- **模块私有组件** → 放在 `app/(module)/components/` 下
- **全局共享组件** → 放在 `components/` 下（如 shadcn/ui）

## 📝 注意事项

1. **禁止跨 Route Group 直接依赖 UI**
   - 不能从 `(dashboard)` 直接 import `(admin)` 的组件
   - 需要通过 `lib/modules/` 暴露的 API 进行通信

2. **业务逻辑只能存在于 Domain 层**
   - API Route、Page 中不得编写业务规则
   - 所有业务逻辑应在 `domain/*.service.ts` 中

3. **模块间通信规范**
   - 通过 `lib/modules/` 暴露模块 API
   - 禁止直接 import 其他模块的 domain 层
